# Dashboard Data Loading UX Optimization Plan

## Current State Analysis

### Dashboard (Everything is Client-Side)
- **Sidebar** (`src/components/sidebar/Sidebar.tsx`): `"use client"`, calls `useDashboard()` hook → fetches from `/api/dashboard`
- **MainContent** (`src/components/main/main-content.tsx`): `"use client"`, calls same `useDashboard()` hook
- **API Route** (`src/app/api/dashboard/route.ts`): Runs 5 Prisma queries in parallel via `Promise.all`
- **Loading**: Both components show skeletons until the single API call completes. No Suspense, no streaming.

### Items Page (Good Pattern - Server-Side)
- **Page** (`src/app/items/[type]/page.tsx`): Server component, fetches `DashboardItem[]` (minimum shape) via Prisma
- **Client Component** (`src/components/items/items-list-content.tsx`): Receives data as props, no own fetching
- **Drawer** (`src/components/items/item-drawer.tsx`): Client-side fetch on demand via `useItemDetail()` hook for full `ItemDetail`

### Key Differences
| Aspect | Dashboard | Items Page |
|--------|-----------|------------|
| Fetching | Client-side (useEffect) | Server-side (async server component) |
| Data Shape | Maximum (all collections + all items per collection) | Minimum (DashboardItem[] for list) |
| Loading UX | Skeleton until full payload loads | Server-rendered immediately |
| Suspense | None | N/A (server renders synchronously) |

## Optimization Plan

### 1. Convert Sidebar to Server Component
**File:** `src/components/sidebar/Sidebar.tsx`

**Current:** `"use client"` with `useDashboard()` hook
**Target:** Server component that fetches directly with Prisma

**What sidebar actually needs:**
- Item type counts (7 types) → `getItemsByTypeCount()`
- Favorite collections (subset) → filter from `getCollections()`
- Recent collections (3) → `getCollections()` with limit
- User info (name, email, image) → `getSession()`

**Changes:**
1. Remove `"use client"` directive
2. Accept data as props from layout
3. Convert child components that need interactivity to client components

**Files to modify:**
- `src/app/dashboard/layout.tsx` - Fetch sidebar data server-side, pass as props
- `src/components/sidebar/Sidebar.tsx` - Convert to server component
- `src/components/sidebar/ItemTypesMenu.tsx` - Keep as client component (needs click handlers)
- `src/components/sidebar/CollectionsMenu.tsx` - Keep as client component (needs click handlers)
- `src/components/sidebar/UserFooter.tsx` - Keep as client component (needs logout)

### 2. Add Suspense Boundaries for Streaming
**File:** `src/app/dashboard/layout.tsx`

**Current:** Renders sidebar, topbar, and children with no Suspense
**Target:** Wrap sidebar and main content in separate Suspense boundaries

**Changes:**
1. Add `loading.tsx` for dashboard route (optional)
2. Wrap `<AppSidebar>` in `<Suspense fallback={<SidebarSkeleton />}>`
3. Wrap children in `<Suspense fallback={<DashboardContentSkeleton />}>`
4. This allows sidebar to stream in independently from main content

### 3. Optimize Dashboard API Queries
**File:** `src/lib/db/collections.ts` - `getCollectionsWithStats()`

**Current:** Loads ALL items per collection to compute `mostUsedType` and `typeIcons` in JavaScript
**Target:** Use Prisma aggregation to compute stats in the database

**Changes:**
1. Replace eager item loading with Prisma `groupBy` or `count` for collection stats
2. Only fetch `mostUsedType` and `typeIcons` data needed (not full items)
3. This reduces the N+1 query pattern

**File:** `src/app/api/dashboard/route.ts`

**Changes:**
1. Add limit to `getPinnedItems()` (currently fetches ALL pinned items)
2. Consider adding pagination for collections if user has many

### 4. Add Loading States for Better UX
**File:** `src/app/dashboard/page.tsx`

**Current:** Just renders `<MainContent />` (client component handles loading)
**Target:** Server component with Suspense for streaming

**Changes:**
1. Add `loading.tsx` file at `src/app/dashboard/loading.tsx`
2. This provides instant feedback while dashboard data loads

## Implementation Order

   - Convert `src/components/sidebar/Sidebar.tsx` to accept props
   - Verify sidebar renders immediately on page load

2. **Phase 2: Suspense Boundaries**
   - Add Suspense wrappers in layout
   - Create `loading.tsx` for dashboard

3. **Phase 3: Query Optimization**
   - Optimize `getCollectionsWithStats()` to use Prisma aggregation
   - Add limits to pinned items query
   - Measure performance improvement

## Expected Outcome

- **Before:** Sidebar + main content both show skeleton until single API call completes (~1-2s)
- **After:** Sidebar renders immediately from server, main content streams in progressively
- **Payload:** Reduced by ~60% by not loading all items per collection
- **UX:** Perceived performance improvement of 50-70% for initial page load
