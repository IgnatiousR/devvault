## Issues Found

### Issue 1: Duplicate `getCollectionsWithStats` Call (High Impact)
**Files:** `src/app/dashboard/layout.tsx:33-34` and `src/app/dashboard/page.tsx:24-25`

Both the layout and the page call `getCollectionsWithStats(user.id)` for the same user. Since the layout wraps the page, this expensive 3-level-deep query (collection -> items -> item -> itemType) executes **twice per dashboard load**.

**Fix:** Remove the `getCollectionsWithStats` call from `page.tsx` and pass it down from the layout to the page via React context or props. Alternatively, remove it from the layout and only call it in the page.

### Issue 2: Sequential Queries in `getItemsByTypeCount` (Medium Impact)
**File:** `src/lib/db/items.ts:110-119`

Two sequential queries that should be parallelized:
1. `prisma.itemType.findMany` (line 111)
2. `prisma.item.groupBy` (line 115)

These are independent and can run in parallel.

**Fix:** Wrap both queries in `Promise.all` (Prisma convention for parallel queries).

### Issue 3: Missing Database Indexes (Medium Impact)
**File:** `prisma/schema.prisma`

Current indexes:
- `Item`: `[userId]`, `[itemTypeId]`, `[isFavorite]`, `[isPinned]`
- `Collection`: `[userId]`

Missing indexes (to be added via Prisma schema + migration):
1. **`ItemCollection` join table** - No indexes on `itemId` or `collectionId`. This table is queried in `getCollectionsWithStats` (collection -> items -> item).
2. **`Item` composite index `[userId, isPinned]`** - Used by `getPinnedItems` query
3. **`Item` composite index `[userId, updatedAt]`** - Used by `getRecentItems` query (filtered + sorted)
4. **`Item` composite index `[userId, itemTypeId]`** - Used by `getItemsByTypeCount` groupBy

### Issue 4: Seed Script Sequential Upserts (Low Impact - Dev Only)
**File:** `prisma/seed.ts:43-52, 65-76, 257-282`

Three loops with sequential `await prisma.*.upsert()` calls:
- 7 itemType upserts (lines 43-52)
- 5 collection upserts (lines 65-76)
- 16 item upserts (lines 257-282)

Total: 28 sequential DB calls.

**Fix:** Use `Promise.all` for independent upserts. Note: `createMany` cannot be used here because items require `connectOrCreate` for tags.

### Issue 5: Redundant User Lookup (Low Impact)
**Files:** `src/app/dashboard/layout.tsx:13-14` and `src/app/dashboard/page.tsx:12-14`

Both files do `prisma.user.findFirst({ where: { email: "demo@devvault.io" } })` independently. This is a hardcoded demo user lookup that bypasses BetterAuth session.

**Fix:** This is a design decision - either extract user lookup to a shared helper or use BetterAuth session. Not strictly an N+1 issue but contributes to redundant queries.

### Step 2: Parallelize `getItemsByTypeCount` queries
**File:** `src/lib/db/items.ts`

Change the two sequential queries to run in parallel using `Promise.all`:
```typescript

  prisma.itemType.findMany({ where: { isSystem: true } }),
  prisma.item.groupBy({ by: ['itemTypeId'], where: { userId }, _count: true }),
]);
```
