# Client-Side Data Fetching with Skeleton Loading

## Architecture Overview

**Current**: Server components fetch data → pass as props to client components
**Target**: Server components render shells → client components fetch their own data via API → skeleton loading states

```
Server (fast shell)              Client (self-fetching)           API
┌─────────────────────┐         ┌──────────────────────┐         │
│ dashboard/layout.tsx│ ──────> │ AppSidebar           │ ──────> │ /api/dashboard
│ (no data fetch)     │         │ (fetches own data)   │         │
├─────────────────────┤         ├──────────────────────┤         │
│ dashboard/page.tsx  │ ──────> │ MainContent          │ ──────> │ /api/dashboard
│ (no data fetch)     │         │ (fetches own data)   │         │
└─────────────────────┘         └──────────────────────┘         │
                                 Shows Skeleton while loading     │
```

## Constraints
- No raw SQL - Prisma only
- Use migrations for schema changes
- Single consolidated `/api/dashboard` API route
- Keep demo user lookup (hardcoded email) for now

## Implementation Steps

### Step 1: Create `/api/dashboard` API route
**New file**: `src/app/api/dashboard/route.ts`

Returns all dashboard data in one call:
```typescript
{
  user: { id, name, email, image },
  collections: CollectionWithStats[],
  pinnedItems: DashboardItem[],
  recentItems: DashboardItem[],
  itemCounts: { totalItems, favoriteItems },
  itemTypesByCount: ItemTypeCount[]
}
```

- Uses existing query functions from `src/lib/db/items.ts` and `src/lib/db/collections.ts`
- Resolves demo user internally
- Returns JSON response

### Step 2: Create shared types file
**New file**: `src/lib/types/dashboard.ts`

Extract and centralize dashboard-related types:
- `DashboardData` (full API response type)
- `User` (sidebar user type)
- Re-export `DashboardItem`, `ItemTypeCount`, `CollectionWithStats`

### Step 3: Create custom hook for dashboard data
**New file**: `src/hooks/use-dashboard.ts`

```typescript
export function useDashboard() {
  // fetches from /api/dashboard
  // returns { data, isLoading, error }
}
```

- Uses native `fetch` + `useState`/`useEffect` (no extra libraries)
- Caches data in state
- Handles loading and error states

### Step 4: Add skeleton components
**New file**: `src/components/ui/dashboard-skeletons.tsx`

Skeleton components matching the layout of:
- `StatsCardsSkeleton` - 4 card placeholders
- `CollectionsSkeleton` - 3 collection card placeholders
- `ItemsSkeleton` - item card placeholders (grid)
- `SidebarSkeleton` - sidebar menu placeholders

All use the existing `Skeleton` component from `src/components/ui/skeleton.tsx`.

### Step 5: Convert `AppSidebar` to fetch own data
**Modify**: `src/components/sidebar/Sidebar.tsx`

- Remove `user`, `collections`, `itemTypes` props
- Use `useDashboard()` hook to fetch data internally
- Show `SidebarSkeleton` while loading
- Keep all existing UI unchanged

### Step 6: Convert `MainContent` to fetch own data
**Modify**: `src/components/main/MainContent.tsx`

- Remove all data props (`collections`, `pinnedItems`, `recentItems`, `totalItems`, etc.)
- Use `useDashboard()` hook to fetch data internally
- Show skeleton loading states while loading
- Keep all existing UI unchanged

### Step 7: Simplify server components
**Modify**: `src/app/dashboard/layout.tsx`
- Remove all Prisma imports and data fetching
- Render shell with `<AppSidebar />` (no props needed)
- Keep `SidebarProvider` and layout structure

**Modify**: `src/app/dashboard/page.tsx`
- Remove all Prisma imports and data fetching
- Render shell with `<MainContent />` (no props needed)
- Keep metadata export

### Step 8: Clean up unused imports
- Remove unused Prisma imports from layout and page
- Remove unused query function imports
- Remove hardcoded user lookup from server components
