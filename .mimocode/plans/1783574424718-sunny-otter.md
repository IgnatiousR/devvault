# Plan: Split Large Components

## Context
Two components exceed the 50-line guideline: `item-drawer.tsx` (689 lines) and `create-item-dialog.tsx` (304 lines). Both contain duplicated type constants and could benefit from extracting logical sections into focused sub-components.

## 2. Split `item-drawer.tsx` → 7 sub-components

New files under `src/components/items/drawer/`:

| File | Lines | Purpose |
|------|-------|---------|
| `drawer-header.tsx` | ~40 | Icon + title + type/language tags (view & edit modes) |
| `drawer-action-bar.tsx` | ~110 | Favorite/Pin/Copy/Edit/Delete (view) + Cancel/Save (edit) |
| `drawer-content-sections.tsx` | ~180 | Description, Content (code/markdown), Language, URL, File/Image preview, Tags, Collections, Details |
| `drawer-skeleton.tsx` | ~30 | Loading skeleton |
| `drawer-delete-dialog.tsx` | ~30 | AlertDialog confirmation for deletion |
| `index.tsx` | ~80 | Main ItemDrawer — orchestrates Sheet + sub-components |
| `types.ts` | ~15 | Shared `EditData` interface and `ItemDrawerProps` |

**Key decisions:**
- `drawer-content-sections.tsx` stays as one file since all sections share the same `item`/`isEditing`/`editData`/`setEditData` props — splitting further would require excessive prop drilling for minimal benefit
- State management (`isEditing`, `editData`, `isSaving`, etc.) and handlers (`handleSave`, `handleDelete`, `handleCancel`) stay in the main `index.tsx` and are passed down as props
- Helper functions (`getItemColorClasses`, `formatDate`) move to `src/lib/item-helpers.ts`

## 3. Split `create-item-dialog.tsx` → 4 sub-components

New files under `src/components/dashboard/create-dialog/`:

| File | Lines | Purpose |
|------|-------|---------|
| `type-selector.tsx` | ~35 | Select dropdown with item type options |
| `content-field.tsx` | ~30 | Conditional CodeEditor/MarkdownEditor based on type |
| `index.tsx` | ~230 | Main dialog — form state, file upload, footer |

**Rationale:** `create-item-dialog.tsx` is only 304 lines. The Type Selector and Content Field are the main candidates for extraction since they have distinct rendering logic. The rest (title, description, URL, tags, file upload, footer) are simple Input/Textarea fields that don't warrant their own files.
